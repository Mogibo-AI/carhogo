## Journey 1: 乗車・Pixel Watch セットアップ

### US-01: Pixel Watch でモニタリングを開始する

**ストーリー**  
ドライバーとして、Pixel Watch の「開始」ボタンを押してモニタリングを起動したい。  
そうすることで、運転中に手動操作なしで生体情報の収集とAWS への送信が始まるようにしたい。

**受け入れ基準**

*主要シナリオ（Gherkin）*
```
Given ドライバーが Pixel Watch アプリを起動している
When 「開始」ボタンをタップする
Then Foreground Service が起動し、Health Services による心拍数・HRV の収集が開始される
And MQTT over TLS で AWS IoT Core への接続が確立される
And 画面のステータスが「モニタリング中」に更新される
And ブラウザアプリがリアルタイム生体情報の受信を開始する
```

*詳細チェックリスト*
- [ ] 「開始」ボタンが画面に表示されている
- [ ] ボタンタップ後、3秒以内にステータスが「接続中」→「モニタリング中」に遷移する
- [ ] 接続失敗時はステータスに「接続エラー」が表示され、バックオフ再接続が自動実行される
- [ ] Foreground 通知（「CarHogo モニタリング中」）が表示される

---

### US-02: Pixel Watch でモニタリングを停止する

**ストーリー**  
ドライバーとして、目的地到着後に「終了」ボタンで安全にモニタリングを停止したい。  
そうすることで、バッテリーの無駄な消費を防ぎ、意図しないトリガーを避けたい。

**受け入れ基準**

*主要シナリオ（Gherkin）*
```
Given モニタリングが稼働中の状態でドライバーが目的地に到着した
When 「終了」ボタンをタップし、確認ダイアログで「OK」を選択する
Then Health Services の収集セッションが停止される
And MQTT 接続が切断される
And Foreground 通知が消える
And 画面のステータスが「停止中」に更新される
```

*詳細チェックリスト*
- [ ] 「終了」ボタンタップ時に確認ダイアログが表示される（誤操作防止）
- [ ] ダイアログで「キャンセル」を選ぶとモニタリングが継続される
- [ ] 停止後、ブラウザアプリの生体情報表示が「—」（無信号）になる
- [ ] 停止後にアプリを閉じても Foreground Service は起動しない

## Journey 2: リアルタイムモニタリング（ブラウザアプリ）

### US-03: ブラウザアプリでリアルタイム生体情報を確認する

**ストーリー**  
ドライバーとして、ダッシュボードのタブレットに表示されたブラウザアプリで自分の心拍数・HRVをリアルタイムに確認したい。  
そうすることで、CarHogoが自分の状態を正しく監視していることを安心して確認できるようにしたい。

**受け入れ基準**

*主要シナリオ（Gherkin）*
```
Given ドライバーが Pixel Watch でモニタリングを開始している
And CarHogo ブラウザアプリにログイン済みである
When 新しい生体データが Pixel Watch から送信される
Then 2秒以内にブラウザアプリの心拍数表示が更新される
And HRV（RMSSD）の値も更新される
And システムステータスが「モニタリング中」と表示される
```

*詳細チェックリスト*
- [ ] 心拍数が数値（BPM）で表示される
- [ ] HRV が数値（ms）で表示される
- [ ] データ受信が途絶えた場合、「接続待機中」のインジケーターが表示される
- [ ] ブラウザアプリは Cognito 認証後にのみ表示される

---

### US-04: ブラウザアプリでシステムのアクション状態を確認する

**ストーリー**  
ドライバーとして、現在アクティブなアクション（SLEEP・ANGER・LATE・アイドル）をブラウザアプリで一目で確認したい。  
そうすることで、CarHogoが今どんな介入をしているのかを把握できるようにしたい。

**受け入れ基準**

*主要シナリオ（Gherkin）*
```
Given ブラウザアプリが表示されている
When SLEEP・ANGER・LATE いずれかのアクションがトリガーされる
Then 2秒以内に該当するアクション状態（例: 「SLEEP アクション実行中」）が表示される
And アクション履歴ログに新しいエントリが追加される
When アクションが終了する
Then アクション状態が「アイドル」に戻る
```

*詳細チェックリスト*
- [ ] アイドル状態・SLEEP・ANGER・LATE の4状態が視覚的に区別できる
- [ ] アクション履歴にタイムスタンプ・アクション種別・継続時間が表示される
- [ ] 最新10件のアクション履歴がリスト表示される

## Journey 3: SLEEPアクション（居眠り防止）

### US-05: 居眠り検知により音声会話が自動的に始まる

**ストーリー**  
ドライバーとして、眠気を自覚する前にCarHogoが検知し、Nova 2 Sonicの音声会話が自動的に始まるようにしたい。  
そうすることで、意識が朦朧とする前に覚醒できるようにしたい。

**受け入れ基準**

*主要シナリオ（Gherkin）*
```
Given ドライバーが高速道路を長距離走行中で、心拍数がベースラインから一定割合低下している
When その状態が一定時間継続する
Then CarHogoが SLEEP パターンを検知し、ブラウザアプリが SLEEP アクション UI に切り替わる
And T0（Pixel Watch データ送信時点）から 5秒以内に Nova 2 Sonic が音声で話しかける
```

*サンプルダイアログ（会話開始）*
> **CarHogo（Nova 2 Sonic）**: 「長距離お疲れ様です。突然ですが、クイズです。日本で一番長い川ってどこか知ってますか？」

*詳細チェックリスト*
- [ ] T0（Pixel Watch データ送信時点）から 5秒以内に Nova 2 Sonic の発話が開始される（詳細は requirements.md「イベントタイムライン定義」参照）
- [ ] ブラウザアプリが SLEEP アクション UI（アクション名・会話状態）に切り替わる
- [ ] ドライバーのマイクが自動的に有効化される
- [ ] 手動トリガー機能（開発者テスト用）でも同じ動作が再現できる

---

### US-06: 音声会話によって覚醒状態を維持する

**ストーリー**  
ドライバーとして、Nova 2 Sonicとの自然な会話を続けることで、停車せずに覚醒状態を保ちたい。  
そうすることで、眠気に自力で抗う努力をしなくても安全に走行を続けられるようにしたい。

**受け入れ基準**

*主要シナリオ（Gherkin）*
```
Given Nova 2 Sonic との SLEEP アクション会話が進行中である
When ドライバーが応答する（またはしない）
Then Nova 2 Sonic は会話を継続し、次の話題やクイズを提供する
And ドライバーの心拍数がベースラインに戻るまで会話を終了しない
```

*サンプルダイアログ（会話継続）*
> **ドライバー**: 「信濃川！」  
> **CarHogo（Nova 2 Sonic）**: 「正解です！さすがですね。ちなみに今日の夕ご飯、もう決めましたか？何か楽しみなことがあると元気出ますよね。」

*詳細チェックリスト*
- [ ] ドライバーが無応答でも会話が一定時間継続される
- [ ] 会話トピックは動的に生成され、繰り返しにならない
- [ ] 会話中はブラウザアプリに「会話中」インジケーターが表示される
- [ ] 会話の内容はドライバーの集中を促すもの（質問・雑談）であり、警告的な内容ではない

---

### US-07: 心拍数の回復により音声会話が終了する

**ストーリー**  
ドライバーとして、心拍数がベースラインに戻ったとき、Nova 2 Sonicとの会話が自然に終了するようにしたい。  
そうすることで、不必要に会話が続かず運転に集中できるようにしたい。

**受け入れ基準**

*主要シナリオ（Gherkin）*
```
Given SLEEP アクション会話が継続中である
When ドライバーの心拍数がベースライン値に回復する
Then CarHogoが回復を検知し、Nova 2 Sonic が会話終了の挨拶をする
And SLEEP アクション UI がアイドル状態に戻る
And アクション履歴に「SLEEP 解消・継続時間」が記録される
```

*サンプルダイアログ（会話終了）*
> **CarHogo（Nova 2 Sonic）**: 「元気になってきましたね。よかったです。あとは安全運転で、楽しい旅を！」

*詳細チェックリスト*
- [ ] 心拍数回復後、30秒以内に会話が自然終了する
- [ ] 会話終了時の Nova 2 Sonic の発話がポジティブなトーンである
- [ ] ブラウザアプリがアイドル状態に戻り、アクション履歴が更新される
- [ ] マイクが自動的に無効化される

## Journey 4: ANGERアクション（怒り・ストレス緩和）

### US-08: 怒り検知により音声会話が自動的に始まる

**ストーリー**  
ドライバーとして、感情が高まる前にCarHogoが検知し、Nova 2 Sonicの音声会話が自動的に始まるようにしたい。  
そうすることで、煽り運転や攻撃的な運転行動に移る前に感情がリセットされるようにしたい。

**受け入れ基準**

*主要シナリオ（Gherkin）*
```
Given ドライバーが静止した状態で（加速度センサーは初期版では固定値を使用）
When 心拍数が短時間で急激にスパイクする
Then CarHogoが ANGER パターンを検知し、ブラウザアプリが ANGER アクション UI に切り替わる
And T0（Pixel Watch データ送信時点）から 5秒以内に Nova 2 Sonic が共感的なトーンで話しかける
```

*サンプルダイアログ（会話開始）*
> **CarHogo（Nova 2 Sonic）**: 「今のはイライラしますよね。気持ちはよく分かります。でも、あなたの冷静さが事故を防いでいます。本当にすごいと思います。」

*詳細チェックリスト*
- [ ] T0（Pixel Watch データ送信時点）から 5秒以内に Nova 2 Sonic の発話が開始される（詳細は requirements.md「イベントタイムライン定義」参照）
- [ ] ブラウザアプリが ANGER アクション UI に切り替わる
- [ ] 会話の最初の一言はドライバーの感情を全肯定・共感するトーンである
- [ ] 手動トリガー機能でも同じ動作が再現できる

---

### US-09: 共感的な音声会話によって感情が落ち着く

**ストーリー**  
ドライバーとして、Nova 2 Sonicが自分の感情を受け止めながら話しかけ続けることで、アンガーマネジメントをCarHogoに代行してもらいたい。  
そうすることで、自分で怒りをコントロールしようとする努力なしに感情が落ち着くようにしたい。

**受け入れ基準**

*主要シナリオ（Gherkin）*
```
Given Nova 2 Sonic との ANGER アクション会話が進行中である
When ドライバーが怒りの感情を言葉で表現する
Then Nova 2 Sonic は怒りを否定せず、全肯定・共感・注意転換の会話を継続する
And ドライバーの心拍数が安定するまで会話を終了しない
```

*サンプルダイアログ（感情受容）*
> **ドライバー**: 「あの割り込みには本当にムカついた！」  
> **CarHogo（Nova 2 Sonic）**: 「そうですよね、あれは誰でも怒ります。でもあなたが冷静に対応したから、今こうして安全に走れているんです。あなたの判断は正しかった。」

*詳細チェックリスト*
- [ ] 会話中は否定・説教・警告的な発話が含まれない
- [ ] ドライバーの発話内容に対してコンテキストに合った応答が返る
- [ ] 会話が「共感 → 肯定 → 注意転換」の流れで進む

---

### US-10: 心拍数の安定により音声会話が終了する

**ストーリー**  
ドライバーとして、心拍数が安定したとき、Nova 2 Sonicとの会話が自然に終了するようにしたい。  
そうすることで、感情が落ち着いた後に運転に集中できるようにしたい。

**受け入れ基準**

*主要シナリオ（Gherkin）*
```
Given ANGER アクション会話が継続中である
When ドライバーの心拍数がベースライン付近まで安定する
Then CarHogoが安定を検知し、Nova 2 Sonic が会話終了の言葉を伝える
And ANGER アクション UI がアイドル状態に戻る
And アクション履歴に「ANGER 解消・継続時間」が記録される
```

*サンプルダイアログ（会話終了）*
> **CarHogo（Nova 2 Sonic）**: 「少し落ち着きましたね。あなたのドライビング、今日もすごく安定しています。このまま安全に行きましょう。」

*詳細チェックリスト*
- [ ] 心拍数安定後、30秒以内に会話が自然終了する
- [ ] 終了メッセージがドライバーを肯定するトーンである
- [ ] ブラウザアプリがアイドル状態に戻る

## Journey 5: LATEアクション（遅刻連絡）

### US-11: 遅刻が自動検知され謝罪メッセージが生成される

**ストーリー**  
ドライバーとして、渋滞などで遅刻確定になったとき、CarHogoが自動的に検知してビジネス向けの謝罪メッセージを生成してほしい。  
そうすることで、焦りながらスマートフォンを操作する必要をなくしたい。

**受け入れ基準**

*主要シナリオ（Gherkin）*
```
Given ドライバーの Google Calendar に予定（開始時刻・場所）が登録されている
And Google Maps API で計算した ETA が予定開始時刻を超過している
When CarHogoが遅刻を検知する
Then Amazon Bedrockがビジネス向け遅刻謝罪メッセージを自動生成する
And ブラウザアプリが LATE アクション UI に切り替わる
```

*詳細チェックリスト*
- [ ] 遅刻検知から 10秒以内にメッセージが生成される
- [ ] Bedrock が未承認の場合、フォールバックテキスト（定型文）が使用される
- [ ] 生成されたメッセージがビジネス敬語で書かれている
- [ ] メッセージには遅刻時間・謝罪・到着予定が含まれる

---

### US-12: Nova 2 Sonic がメッセージを読み上げて送信確認をする

**ストーリー**  
ドライバーとして、Nova 2 Sonicが生成された遅刻メッセージを音声で読み上げて送信可否を確認してくれるようにしたい。  
そうすることで、メッセージ内容をスマートフォンを見ずに確認できるようにしたい。

**受け入れ基準**

*主要シナリオ（Gherkin）*
```
Given Amazon Bedrockが遅刻メッセージを生成した
When Nova 2 Sonic がメッセージを音声で読み上げる
Then ドライバーは送信の可否を音声で応答できる
And Nova 2 Sonic は「このメッセージを送信してもよろしいですか？」と確認する
```

*サンプルダイアログ（読み上げと確認）*
> **CarHogo（Nova 2 Sonic）**: 「おはようございます。現在交通渋滞のため、9時22分頃の到着になる見込みです。ご迷惑をおかけし大変申し訳ございません。——このメッセージを送信してもよろしいですか？」

*詳細チェックリスト*
- [ ] メッセージ全文が音声で読み上げられる（省略なし）
- [ ] 読み上げ後に明確な送信確認の質問がある
- [ ] ドライバーが「はい」「送って」などと応答できる状態になっている
- [ ] ドライバーが「いいえ」「キャンセル」と応答した場合、SMS は送信されない

---

### US-13: ドライバーの音声承認でSMSが自動送信される

**ストーリー**  
ドライバーとして、「送って」と音声で応答するだけで、スマートフォンを操作せずにSMSが送信されるようにしたい。  
そうすることで、運転中の安全を保ちながら謝罪連絡を完了できるようにしたい。

**受け入れ基準**

*主要シナリオ（Gherkin）*
```
Given Nova 2 Sonic がメッセージの送信確認を行っている
When ドライバーが「はい」「送って」などの承認発話をする
Then Amazon SNS（サンドボックス）経由で SMS が送信先に送信される
And Nova 2 Sonic が送信完了を音声で通知する
And ドライバーはスマートフォンに触れない
```

*サンプルダイアログ（送信完了）*
> **ドライバー**: 「はい、送って。」  
> **CarHogo（Nova 2 Sonic）**: 「送信しました。あとは安全運転で、気をつけていってらっしゃい。」

*詳細チェックリスト*
- [ ] ドライバーの承認から 10秒以内に SMS が送信される
- [ ] 送信先は SNS サンドボックスに登録済みの番号のみ（開発フェーズ）
- [ ] 送信完了後に Nova 2 Sonic が音声で確認する
- [ ] ドライバーが拒否した場合、SMS は送信されず LATE アクションが終了する

---

### US-14: ブラウザアプリで送信済みメッセージを確認する

**ストーリー**  
ドライバー（または同乗者）として、ブラウザアプリで送信されたSMSメッセージの内容を確認したい。  
そうすることで、どんなメッセージが送られたかを後から把握できるようにしたい。

**受け入れ基準**

*主要シナリオ（Gherkin）*
```
Given LATE アクションで SMS が送信された
When ブラウザアプリのアクション履歴を確認する
Then 送信済みメッセージのプレビュー・送信先・送信時刻が表示される
And LATE アクション UI がアイドル状態に戻っている
```

*詳細チェックリスト*
- [ ] 送信済みメッセージの全文がブラウザアプリに表示される
- [ ] 送信先番号（マスキングあり）・送信時刻が記録される
- [ ] アクション履歴に「LATE 完了」エントリが追加される
